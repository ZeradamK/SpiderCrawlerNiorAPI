# Payment Tokenization

## Core Philosophy
**"Never store what you don't need to — replace sensitive data with tokens at the earliest possible point."**

## Token Types

### 1. Gateway Tokens (Acquirer/PSP)
- Generated by your payment processor (Stripe `pm_xxx`, Adyen `storedPaymentMethod`)
- Scoped to your merchant account
- Cannot be used across processors
- Simplest to implement

### 2. Network Tokens (Visa VTS, MC MDES, Amex)
- Generated by the card network itself
- Maps to the underlying PAN but is a different number
- **Auto-updates** when card is reissued (no more expired card declines)
- Interchange benefits (Visa: 0.10% reduction)
- Higher approval rates (issuers trust network tokens)

### 3. PCI Tokens (Vault)
- Generated by a PCI-compliant token vault
- Processor-agnostic — use the same token across multiple processors
- Format-preserving option (looks like a card number)

## Network Token Lifecycle

```
1. PROVISION
   Merchant → Network → Issuer
   "I want a token for PAN 4111..."
   ← Token: 4000-xxxx-xxxx-1234
   ← TRID: unique token requestor ID
   ← Cryptogram: dynamic per-transaction

2. TRANSACTION
   Merchant sends token + cryptogram to processor
   Processor → Network: de-tokenize → PAN
   Network → Issuer: authorize with real PAN + TAVV
   
3. LIFECYCLE
   Card reissued → Network auto-updates token mapping
   Card canceled → Network notifies merchant (token status: INACTIVE)
   Merchant deletes CoF → Network deletes token (DEACTIVATED)
```

## Credential on File (CoF) Framework

### Transaction Categories

| Category | Initiated By | Examples |
|----------|-------------|----------|
| CIT (Cardholder-Initiated) | Customer | One-click checkout, saved card purchase |
| MIT (Merchant-Initiated) | Merchant | Subscriptions, installments, no-show charges |
| Recurring | Merchant | Fixed amount, fixed schedule |
| Unscheduled | Merchant | Variable amount, triggered by event |

### CoF Indicators
When using a stored credential, include:
1. **Stored Credential Indicator**: `stored` (first use) or `used` (subsequent)
2. **Transaction ID reference**: Link to the original CIT auth
3. **MIT reason**: `recurring`, `installment`, `unscheduled`, `resubmission`

## Vault Security

### Encryption
- AES-256-GCM for data at rest
- TLS 1.3 for data in transit
- HSM-backed key management
- Key rotation every 90 days

### Access Controls
- Role-based access to token vault
- API key scoping (read-only vs read-write)
- IP allowlisting for vault API access
- Audit logging on every token operation

### PCI DSS Scope Reduction
With tokenization:
- **Without tokens**: Entire application stack is in PCI scope
- **With gateway tokens**: Only the checkout page/form is in scope
- **With network tokens**: Minimal scope (token requestor interface only)

## Migration Guide

### From Raw PAN Storage to Tokens
1. Inventory all PAN storage locations (DB, logs, backups, caches)
2. Provision tokens for all active PANs via batch tokenization API
3. Replace PAN references with token references in application code
4. Update recurring billing to use tokens
5. Purge raw PANs from all storage (including backups after retention period)
6. Update PCI self-assessment questionnaire (SAQ A vs SAQ D)

## Monitoring

- Token provisioning success rate: >99%
- Network token auto-update rate: Track card-reissuance coverage
- Token-to-PAN ratio: Target 100% (no raw PANs in production)
- Expired token cleanup: Monthly sweep for inactive tokens
